### Phase 3 — 자동화 고도화 (핵심 업스킬 구간)

#### Apache Airflow — 워크플로우 자동화

> 이 프로젝트에 Airflow를 적용하는 건 단순한 업그레이드가 아니다.
> "내가 설계한 파이프라인을 실무 수준으로 운영하는 것"이다.

**왜 Phase 3에 Airflow인가?**

Phase 1~2까지는 cron으로 충분하다. 그런데 Phase 3이 되면 파이프라인이 이렇게 복잡해진다:

```
[매일 오전 7시]
fetch_prices ─────────────────────────────────┐
fetch_earnings (분기마다) ────────────────────┤
fetch_capex_data (분기마다) ──────────────────┤
                                               ↓
                              calc_returns → calc_bottleneck_score
                              parse_backlog ──┘
                                    ↓
                           signal_quality_check
                                    ↓
                    ┌───────────────┴───────────────┐
               weekly_report                 signal_log 업데이트
                    ↓                               ↓
              Slack 알림 발송              validation 스케줄 등록
```

cron은 이 의존성을 관리할 수 없다. `fetch_prices`가 실패했는데 `calc_returns`가 돌면 오염된 데이터가 된다.

**Airflow로 얻는 것**

| 기능 | cron | Airflow |
|------|------|---------|
| 태스크 의존성 관리 | ❌ 불가 | ✅ DAG으로 정의 |
| 실패 시 재시도 | ❌ 직접 확인 | ✅ 자동 재시도 |
| 실행 히스토리 시각화 | ❌ 로그 파일만 | ✅ 웹 UI |
| 알림 연동 (Slack, 이메일) | ❌ 별도 구현 | ✅ 기본 제공 |
| 파이프라인 모니터링 | ❌ 불가 | ✅ 실시간 |

**배우는 Airflow 개념**

| 개념 | 뜻 | 이 프로젝트에서의 예시 |
|------|-----|----------------------|
| DAG (Directed Acyclic Graph) | 태스크 간 의존성 그래프 | fetch → calc → report 흐름 |
| Operator | 태스크 실행 단위 | PythonOperator (Python 함수 실행) |
| Schedule | 실행 주기 설정 | `@daily`, `@weekly`, `0 7 * * 1-5` |
| XCom | 태스크 간 데이터 전달 | 수익률 계산 결과를 리포트로 넘기기 |
| Connection | 외부 서비스 연결 | Slack, 이메일 알림 |
| Sensor | 조건 충족 시 다음 태스크 | 데이터 파일이 생성됐을 때만 계속 |

**실무 언어로**: "Apache Airflow 기반 데이터 파이프라인 설계 및 운영 경험"

## 스킬별 실제 적용처 — 이걸 배우면 어디서 쓰는가

> 스킬을 배우는 것보다 "이게 어디에 쓰이는가"를 알면 동기부여도 되고 이력서 언어도 달라진다.

---

### pandas / numpy — 데이터 처리

| 적용처 | 구체적으로 어떤 업무인가 |
|--------|------------------------|
| **증권사 리서치팀** | 섹터별 수익률 비교표 자동 생성, 지표 계산 자동화 |
| **자산운용사** | 포트폴리오 성과 분석, 팩터 노출도 계산 |
| **IT 기업 데이터팀** | 서비스 지표(DAU, 매출) 집계 및 트렌드 분석 |
| **스타트업** | 엑셀로 하던 보고서 자동화 → Python으로 전환 |
| **개인 사이드 프로젝트** | 이 프로젝트처럼 내 투자 분석 자동화 |

**포인트**: pandas는 Excel의 Python 버전이다. Excel을 쓰는 모든 곳에서 pandas가 더 빠르고 자동화가 된다.

---

### yfinance / API 연동 — 외부 데이터 수집

| 적용처 | 구체적으로 어떤 업무인가 |
|--------|------------------------|
| **핀테크 스타트업** | 주가/환율/금리 데이터 자동 수집 파이프라인 |
| **리서치 자동화** | 매일 오전 시장 데이터 업데이트 자동화 |
| **사내 대시보드** | 실시간 KPI 데이터 외부 API에서 가져오기 |
| **개인 트레이딩** | 내 포트폴리오 수익률 자동 계산 및 알림 |

**포인트**: API 연동은 어느 회사든 필요하다. yfinance는 금융 API지만, 패턴은 REST API 어디서든 같다.

---

### cron / 스케줄링 — 자동화 기초

| 적용처 | 구체적으로 어떤 업무인가 |
|--------|------------------------|
| **모든 회사의 데이터팀** | 매일 오전 리포트 자동 발송, 데이터 백업 |
| **운영팀** | 매주 정산 자동화, 매월 마감 보고서 자동화 |
| **개인 업무** | 반복 업무(파일 정리, 이메일 발송) 자동화 |

**포인트**: cron을 쓸 줄 알면 "이거 매일 해줄 수 있어요?"라는 요청을 5분 안에 해결할 수 있다.

---

### Apache Airflow — 워크플로우 자동화 (고급)

| 적용처 | 구체적으로 어떤 업무인가 |
|--------|------------------------|
| **데이터 엔지니어링 팀** | ETL 파이프라인 관리, 데이터 웨어하우스 적재 스케줄링 |
| **ML Ops 팀** | 모델 재학습 파이프라인, 피처 계산 자동화 |
| **금융 데이터팀** | 장 마감 후 자동 계산 → 리포트 생성 → Slack 발송 |
| **이커머스 데이터팀** | 매일 매출 집계 → 재고 알림 → 정산 자동화 |
| **리서치 자동화** | 이 프로젝트처럼 데이터 수집 → 분석 → 리포트 파이프라인 |

**왜 Airflow가 cron보다 강력한가 — 이 프로젝트 예시**

```
[cron 방식 — Phase 1]
매일 7시: run_daily.sh 실행
→ fetch_prices.py 실패해도 모름
→ calc_returns.py 그냥 돌아감
→ 오염된 데이터로 리포트 생성됨
→ 다음날 출근해서야 발견

[Airflow 방식 — Phase 3]
매일 7시: daily_update DAG 시작
→ fetch_prices 태스크 실패 → 즉시 Slack 알림
→ calc_returns 태스크 대기 (의존성 자동 처리)
→ 오염 없음. 실패한 태스크만 수동 재시도
→ Web UI에서 어디서 실패했는지 1초 만에 파악
```

**Airflow가 쓰이는 회사 규모**: 스타트업부터 대기업까지. 데이터팀이 있는 곳이라면 거의 표준 도구다.

---

### Jinja2 / 리포트 자동화 — 템플릿 엔진

| 적용처 | 구체적으로 어떤 업무인가 |
|--------|------------------------|
| **리서치/전략팀** | 주간 보고서 자동 생성 (데이터만 바꾸면 형식 유지) |
| **마케팅팀** | 개인화 이메일 자동 발송 (이름, 수치 자동 삽입) |
| **운영팀** | 계약서, 견적서 템플릿 자동 생성 |
| **개발팀** | 코드 자동 생성, 설정 파일 자동 생성 |

**포인트**: 반복되는 문서 작업을 코드로 만드는 것. "이 보고서 매주 만들어야 해요"를 자동화로 해결.

### 신호 추적 / 검증 루프 — 분석 시스템 설계

이건 기술 스킬이 아니라 **사고방식 스킬**이다. 대부분의 분석가가 빠뜨린다.

| 적용처 | 구체적으로 어떤 업무인가 |
|--------|------------------------|
| **퀀트 투자** | 전략 백테스팅, 신호 정확도 추적 |
| **마케팅 분석** | A/B 테스트 결과 추적, 캠페인 효과 검증 |
| **ML 모델 운영** | 모델 예측 vs 실제 결과 추적 (Model Monitoring) |
| **리스크 관리** | 리스크 지표 임계값 초과 시 알림, 이력 추적 |

**포인트**: "내 분석이 맞았는가"를 추적하지 않으면 개선이 없다. signal_validation.csv 구조 자체가 ML 모델 모니터링과 동일한 패턴이다.

## Airflow 적용 로드맵 (이 프로젝트 기준)

### Step 1 — 로컬 설치 및 기존 스크립트 DAG 변환
```bash
pip install apache-airflow
airflow standalone  # 로컬 개발 서버
```

기존 `run_daily.sh`를 아래처럼 DAG으로 변환:

```python
# dags/daily_update_dag.py
from airflow import DAG
from airflow.operators.python import PythonOperator
from datetime import datetime

with DAG("daily_update", schedule="0 7 * * 1-5", start_date=datetime(2026, 3, 1)) as dag:
    fetch = PythonOperator(task_id="fetch_prices", python_callable=fetch_prices)
    calc  = PythonOperator(task_id="calc_returns", python_callable=calc_returns)
    score = PythonOperator(task_id="calc_score",   python_callable=calc_bottleneck_score)

    fetch >> calc >> score  # 의존성: 순서대로 실행
```
