## 배운 핵심 개념

### FastAPI vs MCP 구조 비교

| 구분 | FastAPI | FastMCP |
|------|---------|---------|
| 앱 초기화 | `app = FastAPI()` | `mcp = FastMCP("...")` |
| 엔드포인트 등록 | `@app.post("/slack")` | `@mcp.tool()` |
| 통신 방식 | HTTP (GET, POST, URL) | MCP 프로토콜 (AI가 tool 호출) |
| 사용자 | 사람 (브라우저) | AI (Claude 등) |
| 핵심 식별자 | URL 경로 | 함수 이름 + Docstring |

둘 다 **"요청 → 함수 실행 → 결과 반환"** 흐름이지만, 호출하는 주체가 다르다.

### 오늘 직접 만든 MCP 서버 (`test_mcpserver.py`)

```python
from fastmcp import FastMCP
import yfinance as yf

mcp = FastMCP("my_stock_mcp")

@mcp.tool()
def get_stock_info(ticker: str) -> str:
    stock = yf.Ticker(ticker)
    info = stock.info
    price = info.get("currentPrice") or info.get("regularMarketPrice", "N/A")
    ...
    return f"현재가: ${price} ..."

if __name__ == "__main__":
    mcp.run()
```

- `@mcp.tool()` = FastAPI의 `@app.get("/...")` 과 동일한 역할 — 함수를 "도구"로 등록
- Claude Desktop에 등록하면 "NVDA 주가 알려줘"라고 말하는 것만으로 이 함수가 호출됨

### GET/POST 설계 방식

**GET — 내가 설계하는 API**

```
GET /stock/samsung           # 종목명으로 조회
GET /stock/samsung?days=7    # 최근 7일치만
GET /stock/samsung?currency=usd  # 달러로 환산
```

- 주소 이름: 내가 설계
- Output 형태: 내가 설계 (숫자 / 문장 / JSON)
- 필터링 조건: 내가 설계 (Query Parameters)

**POST — 외부 서비스가 규칙을 정한다**

```python
@app.post("/slack")   # Slack이 POST로 보내기로 이미 정해뒀음
```

- Slack이 내 서버로 요청을 보내는 구조 → Slack 규칙을 따라야 함
- 주소(`/slack`)도 Slack이 지정

---

### HTTP 메서드의 핵심 원칙

나쁜 예 — 주소에 행동을 구구절절 써놓은 것
```
GET /getStockInfoByName/samsung
```

좋은 예 — 대상만 주소에, 행동은 메서드로
```
GET /stock/samsung
```

> HTTP 엔드포인트는 요청이 들어오는 주소이고, 요청 방식(GET/POST)은 **요청을 보내는 주체**에 따라 결정된다.
